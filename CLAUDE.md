## 1. Golden rules

1. **Content lives in data, never in code.** HTML/JS files contain *logic and layout only*.
   The **source of truth is the per-book Excel workbook in `content/`**; the JSON in `data/**`
   is a **generated artifact** produced by `tools/build_data.py`. To add a question/card/note,
   edit the spreadsheet and re-run the converter — never edit code, and don't hand-edit the
   generated JSON (it gets overwritten on the next build).
2. **The app has no runtime build step or dependencies.** Vanilla HTML/CSS/JS only — no npm,
   bundlers, frameworks, or TypeScript. It must run by serving the folder statically, and it
   reads only the generated JSON in `data/**`. Two **separate, local, offline tools** exist and
   are *not* part of the runtime:
   - `tools/build_data.py` — the **spreadsheet → JSON converter** (Python 3). Run after editing
     content in `content/*.xlsx`; commit the regenerated JSON.
   - **Google Play (TWA) wrapper** — generated with PWABuilder/Bubblewrap; wraps the live PWA
     URL; re-run only on icon/name/start-URL changes, **never** for content edits.
   Neither tool runs on the user's device or blocks static hosting on GitHub Pages.
3. **One design system.** All colours, fonts and spacing come from the CSS variables in
   `assets/app.css`. Never hardcode a hex colour in a page — use the `--read/--recall/--rank`
   tokens (see §3).
4. **Offline-first.** Every new page must be added to the service-worker precache list in
   `sw.js`, and must work with no network once its data is cached.
5. **Everything works on phone and laptop.** Design mobile-first; verify layouts down to a
   360px-wide screen and up to a desktop browser.
6. **Fail gracefully.** If a book/chapter/deck isn't `ready` or a field is missing, show a
   friendly "coming soon" state — never a blank screen or a JS error.
7. **This repo is PUBLIC — never commit private or strategic content.** `adhmukesh-nepal/3r-academy`
   is a public GitHub repo, and **everything at the repo root** (`SPEC.md`, all `*.md` docs) is
   browsable on GitHub even though only `docs/` is served on the website. **Never commit** competitor
   analysis, business/pricing strategy, unreleased roadmap reasoning, revenue numbers, access codes,
   personal data, or any internal discussion. Those belong in **private memory** (`~/.claude/…`) or a
   **gitignored `notes/` folder** — never in a tracked file. When in doubt, don't commit — ask first.

## 2. Folder layout

```
repo root/
  docs/                 # ← GitHub Pages serves THIS folder (the public website)
    index.html          # Home — book grid + track filter
    book.html           # Book page        (?book=<id>)
    chapter.html        # Revision Station  (?book=<id>&ch=<n>)
    flashcards.html     # Flip-card viewer  (?book=<id>&ch=<n>&deck=<id>)
    quiz.html           # MCQ mode          (?book=<id>&ch=<n>&mode=<study|timed|weak|starred>)
    assets/
      app.css           # design system (all shared styles)
      app.js            # shared logic: fetch, render, routing, progress store
    gate.js             # per-book access-code gate
    manifest.webmanifest, sw.js   # PWA manifest + service worker (offline)
    icons/              # icon-192.png, icon-512.png, maskable, favicon
    .well-known/assetlinks.json   # Digital Asset Links (TWA)
    .nojekyll           # let GitHub Pages serve the .well-known folder
    CNAME               # custom domain (do not remove) — 3r.mukeshadhikari.com
    data/               # GENERATED by tools/build_data.py — committed & served, do not hand-edit
      books.json        # master list of books (+ per-book codes)
      <book-id>/book.json, ch<n>.json
  content/              # SOURCE OF TRUTH — author edits these (NOT web-served)
    <book-id>.xlsx      # one Excel workbook per book
  tools/
    build_data.py       # converter: content/*.xlsx → docs/data/**/*.json (NOT web-served)
  SPEC.md, CLAUDE.md, TWA-PLAYSTORE.md   # project docs (NOT web-served)
```

**Naming:** book ids and deck ids are lowercase-kebab-case (`ha-loksewa`, `epi`). Chapter
files are `ch<number>.json`. Keep them consistent — the URLs depend on them.

## 4. Authoring (for the author — edit the spreadsheet, not JSON)

You author content by editing one **Excel workbook per book** in `content/`, then running the
converter. **You never touch JSON or code.**

> **Every-time workflow:** edit `content/<book-id>.xlsx` → run
> `./build.sh` → `git add -A && git commit -m "…" && git push`.
> (`./build.sh` runs the converter in a self-installing local Python env; equivalent to
> `python3 tools/build_data.py`.)
> GitHub Pages redeploys automatically (and the installed Android app picks it up too). The
> converter checks your content and prints plain-language errors *before* writing anything, so
> a mistake can't push a broken page.

### 4.1 Workbook sheets (one workbook per book)

| Sheet | Columns (header row) | Notes |
|---|---|---|
| `Book` | `id, name, board, desc, code, ready, title, edition, tagline, order, category` | one row; `id` is lowercase-kebab (`ph-officer`); `name` = short catalog name, `title` = full book title on the book page; `ready`/`order` control the home grid; `category` = the exam track (`loksewa` / `license` / `entrance`) that drives the home-page filter chips; `ready` = `TRUE`/`FALSE` |
| `Chapters` | `n, title, ready, kind` | one row per chapter. `kind` blank/`chapter` = normal chapter; `kind = test` = a **full-length mock test** (shown in its own "Full-length mock tests" section, opens straight into timed mode). Give tests a distinct `n` (e.g. 101) and put their questions in the `MCQs` sheet with `chapter = 101`. |
| `Subtopics` *(optional)* | `chapter, id, title, desc, ready, order` | one row per subtopic **within** a chapter (a chapter is a big topic, e.g. *Biostatistics*; subtopics are its concepts). `id` = lowercase-kebab, unique within the chapter. When a chapter has ≥1 subtopic row it becomes a **subtopic chooser**, and each subtopic is its own study unit (own notes/decks/MCQs/quiz/progress/ranking). `ready`/`order` behave like chapters. A chapter with **no** subtopic rows behaves exactly as before. |
| `Notes` | `chapter, subtopic, note` | one row per one-liner; `chapter` = the chapter number. `subtopic` = the subtopic `id` (**required** if that chapter uses subtopics; leave blank otherwise). |
| `Flashcards` | `chapter, subtopic, deck_id, deck_name, deck_desc, front, back` | one row per card; rows sharing a `deck_id` form a deck (deck_name/deck_desc taken from the first row of that deck). `subtopic` as above. |
| `MCQs` | `chapter, subtopic, question, correct, option1, why1, option2, why2, option3, why3, option4, why4` | `correct` = which option number (1–4) is right. `subtopic` as above. |
| `Videos` | `chapter, subtopic, title, yt` | leave `yt` blank → "coming soon" card; fill the YouTube id to embed. `subtopic` as above. |

A **book that isn't ready yet** (coming soon) still gets its own workbook with just the `Book`
sheet filled in (`ready = FALSE`) and empty content sheets — that's how it appears greyed-out on
the home grid.

**Rules the converter enforces:**
- **Access code:** each book has its own; case/space/hyphen-insensitive at entry. **Add a random
  suffix** so the code can't be guessed from the book's public title/edition/year — e.g.
  `ADH-<BOOK><ED>-<YEAR>-<random>`, e.g. `ADH-PHO1-2026-7QK9M2`. This matters: the content encryption is only
  as strong as the code, so a code derivable from public metadata (author+book+edition+year) is
  weak. Never commit the code anywhere — it lives only in the spreadsheet.
- **MCQs:** `correct` must be 1–4 and **every** `option`/`why` in use must be filled. Keep each
  `why` to 1–2 sentences; make the wrong ones explain the *specific* misconception (the correct
  one is auto-prefixed with "Correct —").
- **Videos:** `yt` is the id after `watch?v=` in a YouTube URL (e.g. `dQw4w9WgXcQ`).
- Set a book/chapter `ready` to `TRUE` only when it's actually live; otherwise it shows as a
  greyed "coming soon" card.

### 4.2 Generated-output reference (what the converter writes — you don't edit this)

For contributors/Claude verifying the converter, these are the target JSON shapes per sheet:

```json
// books.json entry  ← Book sheet
{ "id":"ph-officer", "name":"Public Health Officer — Loksewa", "board":"PSC · Government job",
  "desc":"Comprehensive Review", "code":"ADHPHO12026", "ready":true }

// chapter file ch6.json  ← Notes / Flashcards / MCQs / Videos sheets (filtered by chapter)
// (this is the FLAT shape — a chapter with NO subtopics)
{ "book":"ph-officer", "number":6, "title":"…",
  "notes":[ "P = I × D. Incidence = new; prevalence = new + old." ],
  "decks":[ { "id":"diseases", "name":"Disease KEY facts",
              "cards":[ ["Cholera stool?", "Rice-water stool, painless, afebrile."] ] } ],
  "mcqs":[ { "q":"The infective form of the malarial parasite is",
             "options":[
               { "text":"Sporozoite", "correct":true,  "why":"Correct — sporozoites from the mosquito's salivary glands infect humans." },
               { "text":"Merozoite",  "correct":false, "why":"Merozoites spread within the human host; they don't initiate infection." },
               { "text":"Gametocyte", "correct":false, "why":"Gametocytes infect the mosquito, not humans." },
               { "text":"Hypnozoite", "correct":false, "why":"Dormant liver stage causing relapse (vivax/ovale), not the infective form." }
             ] } ],
  "videos":[ { "title":"Levels of prevention & immunization", "yt":"" } ] }

// chapter file WITH subtopics ← same sheets, bucketed by (chapter, subtopic)
// The chapter object holds a `subtopics[]`; each subtopic carries its OWN notes/decks/mcqs/videos.
{ "book":"pho-loksewa", "number":1, "title":"Biostatistics", "kind":"chapter",
  "subtopics":[
    { "id":"central-tendency", "title":"Measures of central tendency", "desc":"", "ready":true,
      "notes":[ … ], "decks":[ … ], "mcqs":[ … ], "videos":[ … ] },
    { "id":"dispersion", "title":"Measures of dispersion", "ready":false,
      "notes":[], "decks":[], "mcqs":[], "videos":[] }
  ] }

// book.json chapter entries also expose a subtopic summary (public table of contents):
//   { "n":1, "title":"Biostatistics", "ready":true, "kind":"chapter",
//     "subtopics":[ { "id":"central-tendency", "title":"…", "ready":true }, … ] }
```

The app reads the chapter file and: if it has `subtopics`, `chapter.html` shows a **subtopic
chooser** (open a subtopic via `?book=&ch=&sub=<id>`); otherwise it renders the flat chapter as
before. URLs gain an optional `&sub=<id>`.

## 5. Code conventions (for contributors / Claude)

- **Rendering:** build DOM with small helper functions in `app.js` (e.g. an `el(tag, cls, html)`
  helper — already the pattern in the current `ch5.html`). No template libraries.
- **Routing:** pages read state from query params (`?book=&ch=&deck=`). Centralize parsing in
  `app.js`.
- **Data fetch:** one `loadJSON(path)` helper with a `try/catch` that shows a friendly error
  card; never let a fetch failure blank the page.
- **State/storage keys** (as shipped — namespace any new key here): `3r_keys` (`{bookId: code}`
  unlock map), `progress:<book>:<ch>` (one JSON blob per chapter holding `quiz`, `cards`, `starCards`,
  `starMcqs`, `weakMcqs` — SRS lives **inside** this blob, not a separate `srs:` key), `tr_streak`
  (`{count,last}`) and `tr_goal` (`{day,cards,quizzes}`) for the global streak + daily goal, `theme`.
  **Subtopic progress nests inside the chapter blob** under `p.subs[<subId>]` (same shape as the
  top-level fields) so the key stays 3-segment and account sync is unchanged — use
  `getUnitProg`/`setUnitProg(book, ch, sub)` (a falsy `sub` = the top-level blob).
- **Progress is keyed by CONTENT IDENTITY, not position.** A card's key is `"c"+hash(front)`, an MCQ's
  is `"q"+hash(question)` (`hashStr`, `unitKeys`, `pruneProgress` in `app.js`). So authors may add,
  insert, reorder or delete rows and edit answers/options/explanations **without** disturbing anyone's
  SRS/stars/weak-areas; only rewriting a card's front or an MCQ's question stem resets that one item.
  `pruneProgress` drops keys with no matching content on load (this is what performed the one-time
  reset when the scheme changed from position-based keys). Keep the hash in lock-step if you ever
  compute it elsewhere. Author-facing rules live in `HOW-TO-UPDATE.md`.
- **Accessibility:** buttons are real `<button>`s; cards flippable by keyboard; sufficient
  colour contrast (the palette above already passes).
- **Content is encrypted; codes live only in the spreadsheet.** Chapter content is served as
  AES-GCM ciphertext (`docs/data/*.enc`), keyed by the book's access code (see SPEC §5.5). The
  code is set in the Book sheet and is used only at build time to encrypt — it is **never**
  written to `books.json` or anywhere in the repo. `content/*.xlsx` (plaintext + codes) is
  git-ignored and must be **backed up privately** (it is your master copy, not in the repo).
- **Crypto parameters must stay in lock-step** across `tools/build_data.py`, `docs/gate.js`, and
  `docs/assets/app.js`: PBKDF2-HMAC-SHA256, 200000 iterations, 16-byte per-book salt
  (`sha256("3r-salt:"+id)[:16]`), AES-256-GCM. Change one → change all three, or decryption breaks.

## 6. Deploy & verify

- **Host:** GitHub Pages (deployed from the `docs/` folder), custom domain via `CNAME`
  (`3r.mukeshadhikari.com`). Do **not**
  delete `CNAME` or `404.html`.
- **Local preview:** `cd docs && python3 -m http.server 8080` → open `localhost:8080`.
  (Run the converter from the repo root: `python3 tools/build_data.py`.)
- **Before pushing content:** run `python3 tools/build_data.py` (regenerates `data/**` from
  `content/*.xlsx` and validates it), then open the affected page locally, unlock with the book
  code, and click through the new deck/MCQ/notes.
- **Before pushing code:** run the full verification checklist in `SPEC.md` §8 (laptop flows,
  phone install, offline, Lighthouse PWA, gate isolation).
- **Service worker gotcha:** after changing cached files, bump the cache version constant in
  `sw.js` so users get the update (document the current version at the top of `sw.js`).
- **Google Play (TWA):** the Android app is a TWA wrapping `https://3r.mukeshadhikari.com`,
  generated once with PWABuilder/Bubblewrap. Because it points at the live PWA, **content
  pushed to GitHub Pages appears in the installed Android app automatically** — no re-submission
  needed. Re-package only when the app icon, name, or start URL changes. Requires a one-time
  Google Play developer account ($25) and a signed release keystore (keep the keystore safe —
  losing it blocks future updates).
